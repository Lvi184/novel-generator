# 小说仿写系统 — 深度思路设计

------

## 一、先想清楚：仿写到底在仿什么？

```
一部小说可以拆成这些层次：

┌─────────────────────────────────────────────────┐
│  表层（皮）  → 文字/文风/修辞/语感               │  ← 可以换
│  中层（肉）  → 角色/设定/世界观/具体情节          │  ← 可以换
│  深层（骨）  → 故事结构/节奏曲线/冲突模式/情感弧线  │  ← 要保留
│  核层（魂）  → 主题/价值观/读者爽点设计            │  ← 要保留
└─────────────────────────────────────────────────┘

仿写 = 保留「骨+魂」，替换「皮+肉」
```

### 举个例子让你直觉理解

```
原著《斗破苍穹》:
  骨: 天才陨落 → 隐忍蛰伏 → 逆袭打脸 → 更大的世界 → 循环升级
  魂: "莫欺少年穷"的爽感，阶梯式升级的节奏
  肉: 萧炎、药老、异火、斗气大陆
  皮: 天蚕土豆的文风

仿写后可以变成:
  骨: (保留) 天才陨落 → 隐忍 → 逆袭 → 新世界 → 循环
  魂: (保留) 同样的爽点节奏
  肉: (替换) 都市程序员、AI导师、代码异能、科技公司层级
  皮: (替换) 你想要的文风
```

## 二、完整流程设计

```
                    ┌─────────────┐
                    │  原著小说TXT  │
                    └──────┬──────┘
                           │
            ═══════════════╧═══════════════
            ║    Phase 1: 深度解剖 (原著→骨架)  ║
            ═══════════════╤═══════════════
                           │
                    ┌──────┴──────┐
                    │  结构化骨架   │ ← JSON，人可读可编辑
                    └──────┬──────┘
                           │
                    ┌──────┴──────┐
                    │ 用户审核/微调  │ ← 你觉得哪里不对可以手动改
                    └──────┬──────┘
                           │
            ═══════════════╧═══════════════
            ║   Phase 2: 骨架变形 (旧骨架→新骨架) ║  ← + 你的改写需求
            ═══════════════╤═══════════════
                           │
                    ┌──────┴──────┐
                    │  新大纲/新骨架 │ ← JSON，人可读可编辑
                    └──────┬──────┘
                           │
                    ┌──────┴──────┐
                    │ 用户审核/微调  │ ← 再次确认
                    └──────┬──────┘
                           │
            ═══════════════╧═══════════════
            ║   Phase 3: 血肉生成 (新骨架→新小说)  ║
            ═══════════════╤═══════════════
                           │
                    ┌──────┴──────┐
                    │  完整新小说    │
                    └─────────────┘
```

**关键设计理念：每个 Phase 的输出都是人可读可编辑的 JSON，你随时可以人工介入修改。**

## 三、Phase 1 深度解剖 — 最复杂的一步

### 3.1 为什么复杂？

```
难点: 一本小说可能50万字，LLM上下文窗口最大也就12万字
      而且一次性塞太多，提取质量急剧下降

所以核心问题是: 如何分而治之，又不丢失全局信息？
```

### 3.2 分层提取策略

```
不能一次提取所有东西，要分步骤，每步专注一件事

Step 1: 物理分割
┌──────────────────────────────────────────────┐
│  原著TXT                                      │
│  ↓                                            │
│  按章节标记分割（正则匹配"第X章"）               │
│  ↓                                            │
│  chapters = [{num, title, content}, ...]      │
│                                                │
│  注意点:                                       │
│  - 不是所有小说都有规范章节标记                   │
│  - 需要多种正则兜底                              │
│  - 实在没有标记就按固定字数切                     │
│  - 这步纯代码，不需要LLM                        │
└──────────────────────────────────────────────┘

Step 2: 逐章摘要（Map 阶段）
┌──────────────────────────────────────────────┐
│  每一章单独送给LLM，提取:                       │
│  ├── 本章摘要（500字浓缩）                      │
│  ├── 出场角色列表                               │
│  ├── 关键事件列表                               │
│  ├── 场景/地点                                  │
│  └── 情感基调/紧张度(1-10)                      │
│                                                │
│  为什么逐章？                                   │
│  - 每章通常1万字以内，LLM能处理                  │
│  - 逐章提取信息损失最小                          │
│  - 可以并发处理，速度快                          │
│                                                │
│  输出: chapter_summaries = [{                  │
│    num, title, summary, characters,            │
│    events, location, tension_level             │
│  }, ...]                                       │
└──────────────────────────────────────────────┘

Step 3: 全局合成（Reduce 阶段）
┌──────────────────────────────────────────────┐
│  把所有章节摘要拼起来（摘要够短，拼完能塞进上下文）│
│  然后让LLM做全局分析:                           │
│                                                │
│  3a. 角色全局分析                               │
│  ├── 合并所有章出现的角色 → 去重                  │
│  ├── 分析每个角色的完整弧线（成长/变化）           │
│  ├── 提取角色深层性格（不是表面描述）              │
│  └── 分析角色的叙事功能（为什么需要这个角色）       │
│                                                │
│  3b. 关系网络分析                               │
│  ├── 角色两两关系                               │
│  ├── 关系的动态变化（开始→中间→结局）             │
│  └── 关系对剧情的驱动作用                        │
│                                                │
│  3c. 故事结构分析 ← 这是仿写的核心！              │
│  ├── 整体结构识别（三幕/英雄之旅/多线交织）        │
│  ├── 主线/支线识别与分离                         │
│  ├── 转折点标记（哪些章是关键转折）                │
│  ├── 节奏曲线（紧张度随章节变化的曲线）            │
│  └── 爽点/钩子模式（作者是怎么设计阅读快感的）      │
│                                                │
│  3d. 伏笔系统提取                               │
│  ├── 伏笔列表（在第几章埋的，第几章收的）          │
│  └── 因果链（事件A导致事件B导致事件C）             │
│                                                │
│  3e. 世界观提取                                 │
│  ├── 力量体系/等级体系                           │
│  ├── 社会结构                                   │
│  ├── 地理环境                                   │
│  └── 核心规则/限制                               │
└──────────────────────────────────────────────┘
```

## 四、Phase 2 骨架变形

### 4.1 变形的层次

```
用户输入改写需求:
{
  "new_genre": "都市科技",
  "new_world": "近未来，AI觉醒的世界",
  "tone": "轻松幽默",
  "protagonist_concept": "社恐AI工程师",
  "reference_style": "类似《大王饶命》的吐槽风格"
}

然后分步变形:

Step 1: 世界观映射
┌──────────────────────┬──────────────────────┐
│      原著             │       新作            │
├──────────────────────┼──────────────────────┤
│ 斗气大陆             │  近未来科技都市        │
│ 斗气等级体系          │  AI权限等级体系        │
│ 异火(稀有资源)        │  核心算法(稀有资源)    │
│ 宗门/家族            │  科技公司/开源社区      │
│ 炼药师(稀缺职业)      │  AI训练师(稀缺职业)    │
└──────────────────────┴──────────────────────┘
这步让LLM来做映射，但要人工审核合理性

Step 2: 角色映射
┌──────────────────────┬──────────────────────┐
│ 萧炎(陨落天才)        │  林默(被辞退的天才程序员)│
│ 药老(灵魂导师)        │  废弃服务器里的觉醒AI   │
│ 萧薰儿(青梅竹马)      │  高中同学，现在是CEO    │
│ 纳兰嫣然(退婚对象)    │  前女友，科技媒体人     │
└──────────────────────┴──────────────────────┘
保留叙事功能，替换具体设定

Step 3: 剧情节拍映射（逐个beat）
┌──────────────────────────────────────────────┐
│ 抽象beat: "主角遭遇重大挫败/羞辱"              │
│                                              │
│ 原著具象: 被退婚                              │
│ ↓                                            │
│ 新作具象: 被公司以"能力不足"为由裁员，          │
│          前女友发朋友圈暗讽，前同事群嘲          │
│                                              │
│ 要求: 保持相同的情感强度和叙事功能              │
└──────────────────────────────────────────────┘

Step 4: 细化章节大纲
  把每个 beat 展开成具体的章节大纲
  规定每章:
  - 场景/地点
  - 出场角色  
  - 核心事件
  - 对话要点
  - 情感基调
  - 章末钩子
```

### 4.2 改写质量保证

```
改写后需要自动检查:

┌─────────────────────────────────────────────┐
│  检查项                                      │
├─────────────────────────────────────────────┤
│  ✓ 新设定自洽吗？(世界观逻辑有没有矛盾)        │
│  ✓ 角色映射完整吗？(原著重要角色都有对应)       │
│  ✓ 节奏曲线保留了吗？(不能把高潮变平淡)         │
│  ✓ 因果链完整吗？(A导致B这种逻辑有没有断)       │
│  ✓ 新设定够独特吗？(不能改完还太像原著)         │
│  ✓ 新设定够合理吗？(都市背景动不动毁灭世界？)    │
└─────────────────────────────────────────────┘

用一个专门的LLM调用做这个检查
发现问题 → 反馈给改写步骤重新处理
```

## 五、Phase 3 逐章生成

### 5.1 核心挑战：长文本一致性

```
写第1章的时候一切美好
写到第30章，AI可能已经忘了:
  - 主角的口头禅是什么
  - 第5章提到的那把枪后来怎么了
  - 角色A和角色B上次见面说了什么

这是整个系统最大的技术挑战
```

### 5.2 记忆系统设计

```
写第 N 章时，塞给LLM的上下文这样组织:

┌─────────────────────────────────────────────────────────┐
│                    Prompt 组装                           │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  [固定区] 始终存在，每章都塞                              │
│  ├── 系统人设: "你是一位XX风格的作家..."                   │
│  ├── 世界观摘要: 500字压缩版                              │
│  ├── 全书大纲: 压缩版（章节标题列表+一句话摘要）            │
│  └── 角色卡片: 本章出场角色的档案                          │
│      (性格/说话风格/当前状态/与其他在场角色的关系)           │
│                                                         │
│  [动态区] 每章不同                                       │
│  ├── 前一章完整正文: 用于语言衔接                          │
│  ├── 前3-5章摘要: 近期发生了什么                          │
│  ├── 本章细纲: 要写什么                                   │
│  ├── 相关记忆检索: 和本章内容相关的历史片段                  │
│  │   (用向量检索: 本章大纲 → 搜索相关的历史事件/对话)        │
│  └── 活跃伏笔: 已埋未收的伏笔列表                          │
│                                                         │
│  [约束区] 防止出错                                       │
│  ├── 本章应避免: (已知的容易出错的点)                      │
│  └── 连续性提醒: "注意: 角色A的左手在第5章受伤了还没好"     │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 5.3 每章生成后的更新流程

```
生成第N章正文
    │
    ├──→ 质量检查 (另一个LLM调用)
    │    ├── 和大纲一致吗？
    │    ├── 角色言行符合人设吗？
    │    ├── 有没有和前文矛盾？
    │    └── 不合格 → 带着修改意见重新生成
    │
    ├──→ 信息提取 & 记忆更新
    │    ├── 生成本章摘要 (供后续章节引用)
    │    ├── 更新角色状态 (位置/情绪/持有物品/伤情...)
    │    ├── 记录新事件到时间线
    │    ├── 标记伏笔状态变化
    │    └── 存入向量数据库 (供后续语义检索)
    │
    └──→ 保存正文，进入下一章
```

## 六、技术选型建议（给全栈程序员）

```
┌──────────────┬──────────────────────────────────────┐
│   组件        │  建议                                │
├──────────────┼──────────────────────────────────────┤
│   语言        │  Python (生态最好)                    │
│   LLM API    │  先用 DeepSeek (便宜)                 │
│              │  后期可换 Claude/GPT-4o              │
│   API框架    │  FastAPI (你熟悉的话)                  │
```

## 九、你一定会踩的坑 & 解决思路

### 坑 1：LLM 返回的 JSON 经常格式不对

```
现象:
  - 多了个逗号
  - 用了中文引号
  - 在JSON前后加了"好的，以下是结果："这种废话
  - 返回了markdown格式而不是纯JSON

解决:
┌─────────────────────────────────────────────┐
│                                             │
│  LLM原始返回                                │
│       │                                     │
│       ▼                                     │
│  Step 1: 正则提取 ```json ... ``` 块         │
│       │                                     │
│       ▼                                     │
│  Step 2: 修复常见问题                        │
│       ├── 替换中文引号 → 英文引号             │
│       ├── 移除尾部多余逗号                    │
│       ├── 移除注释                           │
│       └── 尝试截取第一个 { 到最后一个 }        │
│       │                                     │
│       ▼                                     │
│  Step 3: json.loads() 尝试解析               │
│       │                                     │
│       ├── 成功 → 返回                        │
│       └── 失败 → 把报错信息连同原文            │
│                  再丢给LLM说"请修复这个JSON"   │
│                  (retry机制, 最多3次)          │
│                                             │
└─────────────────────────────────────────────┘

这个JSON解析器要做得非常健壮
它会在整个系统中被调用几百次
```

### 坑 2：提取结果的质量参差不齐

```
现象:
  - 有些角色被遗漏
  - 关系提取不全
  - 剧情摘要过于笼统, 丢失关键信息
  - 不同批次提取的结果有冲突

解决:
┌─────────────────────────────────────────────┐
│                                             │
│  策略1: 多次提取 + 合并去重                   │
│  ├── 同一段内容, 用不同prompt提取2-3次        │
│  ├── 合并结果, 去掉重复项                     │
│  └── 信息更丰富, 遗漏更少                     │
│                                             │
│  策略2: 交叉验证                              │
│  ├── 提取完角色后, 再用另一个prompt问:         │
│  │   "以下角色列表是否完整？有无遗漏？"         │
│  └── 补充遗漏                                │
│                                             │
│  策略3: 人工兜底                              │
│  ├── 提取结果保存为JSON                       │
│  ├── 提供给用户审核                           │
│  ├── 用户可以手动修改/补充                     │
│  └── 修改后的版本进入Phase 2                  │
│                                             │
│  记住: 不要追求完全自动化                      │
│  人工审核一次, 效果提升巨大                     │
│                                             │
└─────────────────────────────────────────────┘
```

### 坑 3：生成的正文角色性格不稳定

```
现象:
  第3章: 主角是个社恐, 说话结巴
  第7章: 突然变成演说家, 口若悬河
  第12章: 又变回社恐了

原因:
  LLM没有真正"理解"角色, 它只是在做文本生成
  如果prompt里没有足够强调角色特点, 它就会漂移

解决:
┌─────────────────────────────────────────────┐
│                                             │
│  方法1: 角色卡片强约束                        │
│  每次生成都在prompt里放角色卡片:               │
│  """                                        │
│  [角色: 林默]                                │
│  - 核心性格: 极度社恐, 和人说话会紧张          │
│  - 说话特点: 经常说一半就停, 用"那个..."开头    │
│  - 绝对不会: 主动找陌生人搭话, 发表长篇演说     │
│  - 内心活动: 很丰富但说不出来                  │
│  """                                        │
│                                             │
│  方法2: 对话后校验                            │
│  生成完一段对话后, 用另一个LLM调用检查:         │
│  "以下对话是否符合角色人设？如不符合请指出"      │
│  不符合 → 重新生成那段对话                     │
│                                             │
│  方法3: Few-shot 示例                        │
│  在prompt里给几段该角色的对话范例               │
│  让LLM模仿这个"语气"                         │
│                                             │
└─────────────────────────────────────────────┘
```

### 坑 4：章节之间衔接生硬

```
现象:
  第5章结尾: 主角走进了森林
  第6章开头: 主角在城市里吃饭 (???)

  或者更隐蔽的:
  第5章结尾: 紧张的战斗刚打完
  第6章开头: 语气突然变得很轻松, 像换了个作者

解决:
┌─────────────────────────────────────────────┐
│                                             │
│  方法1: 上一章末尾原文输入                     │
│  把上一章最后500-1000字原文直接塞进prompt       │
│  明确指令: "请从以下场景自然衔接, 继续书写"     │
│                                             │
│  方法2: 衔接段专门生成                        │
│  先生成本章开头的200字"过渡段"                  │
│  确认衔接自然后再生成后续正文                   │
│                                             │
│  方法3: 全章生成后检查首尾                     │
│  自动检查: 上一章末尾 + 本章开头 是否连贯       │
│  不连贯 → 重新生成本章开头部分                  │
│                                             │
└─────────────────────────────────────────────┘
```

### 坑 5：API 成本失控

```
现象:
  分析一本50万字的小说, 发现花了200块API费

原因:
  - 每章都用最贵的模型
  - 重试太多次
  - prompt太长, 塞了太多无关信息
  - 没有做缓存, 重跑时重复调用

解决:
┌─────────────────────────────────────────────┐
│                                             │
│  1. 缓存! 缓存! 缓存!                        │
│  ├── 每次LLM调用, 缓存 (prompt_hash → result)│
│  ├── 相同输入不重复调用                       │
│  └── 用SQLite或JSON文件缓存都行               │
│                                             │
│  2. 分级模型                                  │
│  ├── 提取/摘要: 用最便宜的模型                 │
│  ├── 正文生成: 用中档模型                      │
│  └── 只在关键创意环节用最贵的                   │
│                                             │
│  3. 先小规模测试                              │
│  ├── 先只处理前5章, 看看效果                   │
│  ├── 满意了再跑全书                           │
│  └── 别上来就处理200章                        │
│                                             │
│  4. 控制prompt长度                            │
│  ├── 摘要尽量精简                             │
│  ├── 只塞相关信息, 不要"以防万一"全塞           │
│  └── 监控每次调用的token数                     │
│                                             │
│  5. 成本估算器                                │
│  ├── 跑之前先估算: "本次预计调用X次, 费用约Y元" │
│  └── 让用户确认后再执行                        │
│                                             │
└─────────────────────────────────────────────┘
```

### 坑 6：原著太长，提取丢失全局信息

```
现象:
  原著1000章, 每3章一批, 需要333次LLM调用来提取剧情
  最后合并时, LLM也塞不下333条剧情摘要
  导致全局结构分析质量差

解决: 递归摘要 (层层压缩)
┌─────────────────────────────────────────────┐
│                                             │
│  Level 0: 原文 (每章 ~1万字)                  │
│       ↓  逐章摘要                            │
│  Level 1: 每章摘要 (~500字/章)                │
│       ↓  每10章合并再摘要                     │
│  Level 2: 每10章摘要 (~800字/组)              │
│       ↓  每5组合并再摘要                      │
│  Level 3: 每50章摘要 (~1000字/组)             │
│       ↓  全部合并                            │
│  Level 4: 全书摘要 (可能就几千字了)             │
│                                             │
│  做全局分析时用 Level 3-4                     │
│  做细节分析时回查 Level 1-2                    │
│                                             │
│  本质就是: 信息金字塔                          │
│  越往上越精炼, 但损失细节                      │
│  需要细节时再往下查                            │
│                                             │
└─────────────────────────────────────────────┘
```
